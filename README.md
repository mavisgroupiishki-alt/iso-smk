## v57 убран риск дублирования сообщения в двух каналах

Клиент получил один и тот же текст и в Telegram, и в Viber после одного нажатия кнопки автопилота. Причина: при HTTP 500 от Wazzup сервер сам повторял тот же запрос через секунду — если первая попытка реально доставила сообщение, а Wazzup просто вернул ошибку с опозданием (известная у них особенность), повтор отправлял дублирующее сообщение. Убрана повторная попытка внутри одного канала при 500 — теперь при ошибке сразу решается, использовать ли запасной канал (Viber), без слепого повтора первого. Если запасной канал всё же сработал после ошибки 500 у основного — в отчёте автопилота теперь явно появится предупреждение проверить вручную, не пришло ли клиенту дублирующее сообщение.

## v54 живой бот в Wazzup-чате (пилот, сделка 34946)

Большая новая фича: ассистент теперь может сам, в реальном времени, отвечать клиенту в Wazzup-чате без нажатия кнопки в Bitrix — настоящий живой автоответчик, а не разовый скрипт по кнопке.

## Как это работает

1. Клиент пишет сообщение в Telegram/Wazzup.
2. Wazzup присылает нашему серверу вебхук на `/api/wazzup/webhook`.
3. Сервер проверяет: это тестовый номер (`LIVE_CHAT_TEST_PHONE`)? Если нет — игнорирует.
4. Отдельный быстрый ИИ-классификатор решает: можно ответить автоматически, или нужен живой человек (жалоба, конфликт, чувствительная тема, неясный вопрос). По умолчанию при любой неопределённости — классификатор выбирает "нужен человек", это сделано специально консервативно.
5. Если безопасно — ИИ генерирует ответ (без упоминания названия мессенджера, как и в обычном автопилоте) и сразу отправляет его клиенту через Wazzup.
6. Если нужен человек — ассистент НЕ отвечает клиенту сам, создаёт срочную (приоритет "высокий") задачу Кристине с полным текстом сообщения клиента и причиной, почему бот не ответил сам.
7. Вся переписка (входящие и исходящие) логируется как комментарии в таймлайне сделки — это используется как память диалога для каждого следующего ответа.

## Что нужно сделать для запуска (один раз)

**Шаг 1.** Создать входящий вебхук в Bitrix24: раздел "Разработчикам" → "Входящий вебхук". Права: CRM (crm), Задачи (task), Пользователи (user), Диск (disk). Скопировать готовый URL вида `https://mavisgroup.bitrix24.by/rest/2110/xxxxxxxx/`.

**Шаг 2.** Добавить в Render Environment:
```
BITRIX_WEBHOOK_URL=https://mavisgroup.bitrix24.by/rest/2110/xxxxxxxx
LIVE_CHAT_ENABLED=true
LIVE_CHAT_TEST_PHONE=375333315898
LIVE_CHAT_TEST_DEAL_ID=34946
WAZZUP_CRM_KEY=придумай любую длинную случайную строку, например mavis-secret-2026-xyz
```
`WAZZUP_CRM_KEY` — это секрет для проверки, что вебхук реально пришёл от Wazzup, а не от случайного запроса; нужно будет указать то же самое значение в настройках Wazzup на шаге 3.

**Шаг 3.** Зарегистрировать вебхук в Wazzup — это можно сделать одним запросом (попроси меня выполнить, когда будет готов `WAZZUP_API_KEY`, или сделай вручную через поддержку Wazzup, указав URL `https://<ваш-домен-на-render>/api/wazzup/webhook`).

После этого живой бот активен только для номера 375333315898 / сделки 34946 — остальные сделки не затрагиваются.

## v51 диагностика Wazzup минимальным payload (без поддержки/без Viber)

## v50 нашёл корневую причину: \b не работает с кириллицей

Это была главная причина, почему проверка передачи "тупила" с самого начала, а не отдельные мелкие баги.

**Корневая причина:** в JavaScript regex `\b` (граница слова) определяется через переход между "словесными" (`\w` = только `[A-Za-z0-9_]`) и "не словесными" символами. Кириллические буквы для движка regex — не словесные символы с обеих сторон, поэтому `\b` вокруг русских слов в принципе никогда не находит границу. Проверено: `/\bперевести\b/i.test('перевести')` — `false`, даже на точном изолированном слове. Это значит большинство паттернов проверки передачи (`\b(перевести|...)\b`, `\b(специалист|...)\b`, `\b(пошлин|...)\b` и так далее) никогда не срабатывали на русском тексте, независимо от того, что реально было сказано или написано. Все кириллические regex-паттерны в файле переписаны без `\b` (15+ паттернов в блоке REQUIRED_ITEMS, плюс несколько в detectProductProfile, которые отвечали за определение типа продукта — `\b(спк|стк)\b`, `атт\b`).

**Вторая причина (КП-файл):** виджет "Файлы по клиенту" в карточке сделки физически хранит файлы в сущности КОМПАНИИ, а не в самой сделке — выяснилось по присланному скрину (КП/счёт/договор лежат в company timeline, при этом ни одно файловое поле сделки не заполнено). Раньше код проверял только timeline сделки. Теперь дополнительно собираются комментарии и файлы компании (`crm.timeline.comment.list` с `ENTITY_TYPE: 'company'`).

**Третья причина (сроки в разговорной речи):** старый regex для "что обещано по срокам" требовал цифру прямо перед словом "недел" (`\d+\s*недел`), но в реальной речи говорят "через **две** недели" — словом, не цифрой. Добавлены текстовые числительные (одну/две/три/четыре/пять/пару/несколько).

Дополнительно: убран нестандартный заголовок `X-Api-Key` из запросов к Wazzup (не помогло с HTTP 500 — судя по идентичному payload в последнем прогоне, проблема не в заголовках). Гипотеза с `chatId` для уже существующего Telegram-диалога остаётся актуальной и требует вебхука — отдельная задача.

## v49 нашёл реальную причину расхождения с ручной отправкой

Ты прислала скрин: твои ручные сообщения этой клиентке в Telegram уходят из самого Bitrix24 (вкладка WAZZUP, отправитель "Bitrix 24 • Анна Жихарева"), а не из веб-интерфейса Wazzup24 напрямую. Это другой путь, чем прямой вызов нашего сервера к `api.wazzup24.com/v3/message`, и помогло найти конкретную ошибку с нашей стороны:

**Нашёл и исправил:** наш сервер отправлял Wazzup два заголовка авторизации одновременно — `Authorization: Bearer {key}` И нестандартный `X-Api-Key: {key}`. По официальной документации Wazzup `/v3/message` ожидает только `Authorization: Bearer`, заголовка `X-Api-Key` в их схеме нет вообще. Лишний заголовок убран из обоих мест (отправка сообщения и получение списка каналов).

**Гипотеза, которая требует отдельной доработки, не точечного фикса:** у этой клиентки уже есть активный диалог в Telegram с реальным `chatId`, привязанным к разговору. Мы каждый раз отправляем `phone` вместо `chatId`, потому что не знаем настоящий `chatId` — единственный надёжный способ его получить — через вебхук Wazzup на входящие сообщения (тогда Wazzup сам присылает `chatId` при каждом сообщении от клиента, и можно сохранить его в Bitrix). Без вебхука нет способа узнать `chatId` существующего диалога, не угадывая. Это отдельная фича (нужен публичный URL для приёма вебхуков, хранение `chatId` по контакту), не однострочный фикс — обсудим отдельно, если проблема не уйдёт после исправления заголовков.

## v48 проверка передачи была слепой к файлам, звонку и продукту

Разобрал жалобу «ИИ-ассистент тупит» по передаче — нашлись три реальные причины, не связанные с самим ИИ (он просто честно докладывал то, что ему передал код):

1. **КП-файл не находился.** `crm.timeline.comment.list` вызывался без `select: ['FILES']`, поэтому прикреплённые файлы в комментариях вообще не приходили в ответе API — даже если КП реально прикреплено. Теперь поле `FILES` запрашивается явно, имена прикреплённых файлов попадают в текст для проверки, и реальный файл КП (по расширению .pdf/.docx/.xlsx или ключевым словам в названии) засчитывается как найденный.
2. **Канал связи проверялся по тексту, а не по полю сделки.** Детектор искал слова «wazzup/telegram/viber» в комментариях, хотя реальный канал хранится в отдельном поле сделки (`PREFERRED_CONTACT_FIELD_CODE`), которое уже использовалось для отправки сообщений, но не для проверки передачи. Теперь проверка читает то же самое поле напрямую.
3. **Список из 11 требований был общий для всех продуктов.** «Кого подбирает MAVIS» и «средства измерений» — это требования для СТК/СПК, неприменимые к чистой аттестации организации. Список теперь зависит от продукта сделки: для `company_attestation` эти два пункта не запрашиваются вообще.
4. **Проверка передачи не видела сам звонок.** Раньше «Проверка передачи» в автопилоте выполнялась до расшифровки звонка и проверяла только то, что зафиксировали продажи заранее — поэтому пункты, закрытые прямо в звонке (сроки, кого переводить на должность), ошибочно показывались как пробелы. Порядок шагов автопилота изменён: сначала звонок расшифровывается, потом передача проверяется уже с учётом расшифровки — это итоговая оценка, а не только досье от продаж.

Также по факту с Wazzup Telegram (`HTTP 500` с пустым телом даже при правильном payload): это совпадает с документированной у Wazzup ошибкой `MESSAGES_CAN_NOT_ADD` («непредвиденная ошибка сервера»), которая может возникать и при полностью корректном запросе. Дальше это можно прояснить только через их поддержку (отвечают за 5 минут в Telegram-боте @WazzupSupportbot или WhatsApp +371 25893011) с указанием `crmMessageId` неудачного запроса — без доступа к их внутренним логам разница между «у нас баг» и «у них баг» технически не определяется.

## v47 запасной канал + видимый payload ошибки

После v46 Wazzup Telegram всё ещё иногда падает с `Internal Server Error` и пустым телом ответа, хотя ручная отправка из самого Wazzup24 в этот же момент работает. Если после этого Wazzup Telegram продолжит падать и для Viber-фоллбека тоже — это сигнал писать в поддержку Wazzup с конкретным временем запроса, потому что дальнейшая диагностика без доступа к их внутренним логам ограничена.
